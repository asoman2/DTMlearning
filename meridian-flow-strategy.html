For UX team review · Not an implementation

From a single actor
to a collaborative programme

The Hartwell flow works because it assumes one person, one file, one go-live. Meridian breaks every one of those assumptions simultaneously. This document defines the strategy for a migration flow that handles 14 personas, 10 QBDT files, 3 legal entities, and 3 staggered go-live waves — without collapsing under the weight of its own complexity.

01 — Navigation architecture — 4 layers, not 2
Foundation decision

The Hartwell flow has two navigation layers: Stage (5 stages) and Step (steps within a stage). This architecture fails for Meridian because the programme itself has structure above the stage level — three waves of go-live with dependencies between them — and step-level content has a persona dimension that Hartwell never needed. The Meridian flow needs four explicit layers, each with its own navigation treatment and its own purpose.

Layer 1 (Programme) is new and critical. This is the view the CFO and Programme Manager live in. It shows all three waves simultaneously — like a Gantt view but expressed in the journey language of the flow — with dependency arrows showing why Wave 3 cannot begin until Wave 1 has 30 days of stable operation. No equivalent exists in the Hartwell flow. This is the most important new component to design first.
Layer 2 (Wave) is where Meridian's complexity lives. The same stage structure (Discovery → Concept → Data prep → Migration → Validation → Go-live) repeats across all three waves — but Wave 3 is not Wave 1 with more states. It is fundamentally more complex: union CBAs, certified payroll, Washington L&I. The stage structure is the same; the step content within each stage must reflect the additive complexity of each wave. This is not a copy-paste — it is a layered inheritance model.
02 — The persona dimension — 14 actors, 4 view types
Design challenge

The hardest design problem in the Meridian flow is persona representation. Not every step is relevant to every persona. A field supervisor does not need to see the CoA consolidation steps. The Controller does not need to see the field crew direct deposit orientation. But the flow must feel like a coherent programme — not 14 separate journeys that happen to share a database. The solution is a single unified flow with a persona filter layer: the primary view shows all steps; activating a persona filter dims non-relevant steps and highlights the active persona's responsibilities.

Personas are grouped into 4 view types based on their relationship to the migration flow — not their seniority or their job title.

03 — Collaboration patterns — 4 types, each with its own UI treatment
New design language

The Hartwell flow has no language for collaboration — every step has one actor. Meridian's migration has four distinct types of collaboration moment, each with different cognitive and process implications. These cannot all look like a regular step card. They need a distinct visual treatment that signals: "this is a handoff / this is a convergence / this is a parallel track / this is an expert consultation."

Pattern 1 · Sequential Handoff
Persona A completes → Persona B receives and continues
The most common collaboration type. One persona finishes their portion of a step and the responsibility transfers to another. The transfer is explicit — Persona B receives a notification and sees the output of Persona A's work before continuing. The step is not complete until Persona B has accepted the handoff.
"Controller completes CoA consolidation → sends to Karen Alvarez (CPA) for opening balance sign-off → CPA reviews and confirms → Controller is notified and Stage 4 unlocks."
Visual: split step card with → arrow between personas Status: A working → A complete → B reviewing → B confirmed
Pattern 2 · Parallel Workstream
Two personas work simultaneously — outputs merge at a gate
Two or more personas execute independent work concurrently. Their outputs are independent but must both be complete before the programme can advance. The UI shows both workstreams simultaneously with a visual merge point at the gate. Critically: delay by either persona blocks both.
"IT Director builds LCPtracker IES integration (Months 2–7) in parallel with Payroll Manager mapping union CBA pay items (Months 4–6). Both must be complete before Wave 3 go-live can be cleared."
Visual: two parallel swim lanes converging at a gate node Status: both tracks show independent progress bars
Pattern 3 · Gate Convergence
Multiple personas must all sign off before the programme advances
A critical gate that requires explicit confirmation from multiple named personas before any work proceeds. Each signatory has a distinct domain responsibility for their sign-off. One missing sign-off blocks all others from proceeding — no partial approval. The gate card shows named individuals, their status, and what specifically they are confirming.
"Wave 1 go-live gate: CFO (business approval) + Controller (financial accuracy) + Payroll Manager (YTD validation) + Karen Alvarez CPA (opening balances) + Branch Managers TX/AZ/GA/NC (local readiness) — all 7 must confirm."
Visual: gate card with named avatar row — each showing pending/confirmed Behaviour: gate unlocks only when last confirmation received
Pattern 4 · Expert Consultation
A specialist is brought in for a specific review — then exits
A persona who is not part of the day-to-day programme is brought in for one specific expertise-dependent review. They receive a purpose-built package (the Gate-only view), review it, confirm or flag, and exit. They do not navigate the full flow. Their involvement is time-boxed and defined. Examples: Union Steward reviewing fringe benefit configuration; external auditor reviewing WIP schedule.
"Union Steward (Local 520, NY) receives a secure link to a gate-only view showing the IES fringe benefit configuration for their agreement. Confirms rates match CBA. Returns to normal work. No IES login required."
Visual: consultation card with external badge + email-accessible link Timeout: 48-hour response window with escalation path
04 — Wave structure — additive complexity, not repetition
Inheritance model

The three go-live waves share the same stage structure — but they are not copies of each other. Wave 1 is the baseline. Wave 2 inherits Wave 1 and adds specific complexity for Florida, Colorado, and Ohio. Wave 3 inherits Wave 2 and adds union, certified payroll, and Washington L&I. The UI must make this additive complexity visible — a Wave 3 step should look visibly more complex than the same step in Wave 1, because it is. A "Wave complexity adds" section within each step shows what is different in Wave 2 or Wave 3 compared to the baseline.

Wave 3 has a blocking dependency on Month 2 that the flow must surface immediately. The LCPtracker certified payroll integration must begin in Programme Month 2 — not Month 10. This is not a Wave 3 step. It is a Wave 0 (programme setup) step that is a prerequisite for Wave 3. The flow must show this cross-wave dependency visibly: a red thread connecting the LCPtracker step in Wave 0 to the Wave 3 go-live gate, with a countdown showing how much time remains before the integration start date makes the Wave 3 timeline impossible.
05 — Wave 0 — the pre-wave that makes everything else possible
Does not exist in Hartwell

Wave 0 is an entirely new concept with no equivalent in the Hartwell flow. It is the programme-level setup work that must happen before any wave's Discovery stage can begin. It crosses all 10 states, all 14 personas, and all 3 legal entities simultaneously. Many of its steps are blocking for Wave 1, some are blocking for Wave 3, and the LCPtracker integration must be treated as a Wave 0 step that begins in Month 2 regardless of when Wave 3 goes live.

08 — Design principles — what must be true of this flow
Non-negotiable
P.01
Coherent programme, not 14 parallel journeys
The persona filter shows each person their relevant steps — but the underlying structure is always one unified programme. Nobody should feel like they are using a different product from their colleagues. The CFO and the field foreman are both in the same migration journey — just at different levels of resolution.
P.02
Every collaboration moment has a named person, not a role
Gate convergence and sequential handoff cards must show "Karen Alvarez, Mitchell & Park LLP" — not "External CPA." Named individuals create accountability that role labels do not. The flow must be configurable with real names at the start of the programme and those names must appear throughout.
P.03
Complexity is visible but not overwhelming
Wave 3's union and certified payroll complexity must be visible in the step cards — but collapsed by default. A payroll manager viewing Wave 3 can see that this step is more complex than Wave 1 without being overwhelmed by every detail before they need it. Wave complexity adds are collapsed; a single sentence signals the addition; expansion reveals the detail.
P.04
Gates are consequences, not checkboxes
Every gate in the flow must show what happens if it is not cleared — not just that it is "required." "If LCPtracker integration does not begin by Month 2, Wave 3 moves from Month 12 to Month 18." The consequence is the motivation. A gate without a named consequence is a checkbox that people talk themselves into skipping.
P.05
The flow speaks QBDT first, IES second — in every translation
Every concept bridge, tooltip, and mapping table must frame the IES concept in QBDT language first. The 10-state programme has people who have used QBDT Enterprise for 5–15 years. Their mental model is an asset, not a barrier. "Projects (like your Customer:Jobs)" always before "Projects." This is especially critical for Wave 3 personas who have the most complex mental model transitions.
P.06
The WA L&I manual process is a designed solution, not a gap
Washington State's L&I workers' comp system has no IES equivalent. This will never be automated. The flow must present the manual process as a permanent, documented, designed workflow — not as a missing feature or a future fix. Tone: "IES cannot automate this. Here is the process Meridian will use. It is owned, calendared, and requires no IES configuration." Not a TODO. A solution.
09 — Build sequence — what to design first
Recommended order

Not all 12 new components are equally important or equally complex to build. The following sequence prioritises by two criteria: which components are blocking (other components cannot be built until they exist) and which components have the highest decision risk (wrong design requires rebuilding the most). The programme dashboard and the gate convergence card are the two highest-priority, highest-risk components in the entire build.

01
Programme Dashboard (Layer 1)
The CFO's view. Shows all 3 waves simultaneously with cross-wave dependency threads and go/no-go gate status. Design this first because it defines the information architecture that all other layers must conform to. If the Programme Dashboard's mental model is wrong, every layer beneath it inherits the error. This is the highest-risk design decision in the entire project.
Design risk: High Blocks: Wave Selector
02
Gate Convergence Card (Layer 4)
The most frequently appearing new component in the flow. Every major programme gate uses it: Wave 1 go-live gate (7 signatories), Wave 3 certified payroll gate (CPO + PS + Union Stewards), QBDT decommission gate (CFO + Controller + CPA). If the gate convergence card is poorly designed, every critical moment in the programme feels bureaucratic rather than clear. Design it second because it must be right before Wave 0 steps can be fully specced.
Design risk: High Blocks: Wave 0 content
03
Persona Filter Bar + Persona Avatar Chip (Layer 3–4)
The persona filter is the primary navigation mechanism for the 8 deep-domain-view personas. Design the filter bar and avatar chip system before writing any step content — because the persona tagging system determines how steps are authored. Every step must be tagged to its personas before the filter can work. If the tagging system changes after steps are written, every step must be reauthored.
Design risk: Med Blocks: Step authoring
04
Wave 0 steps — content before UI
Write all Wave 0 step content (multi-entity architecture decision, 10-file AI scan, CoA design, LCPtracker integration start, CBA extraction, BI tool selection) before finalising the UI that hosts them. Wave 0 is the most unfamiliar territory — it has no equivalent in the Hartwell flow and its steps involve the most complex collaboration patterns. Getting the content right informs the UI. Getting the UI right without content risks designing for the wrong use cases.
Design risk: Med Informs: LCPtracker cross-wave thread
05
Wave Complexity Adds section (Layer 4)
The component that makes Wave 3 visibly different from Wave 1. Design the collapsed/expanded treatment and the content schema for waveAdds before writing Wave 2 and Wave 3 step content. The design must answer: how much does a collapsed Wave 3 complexity indicator show? (Answer: one sentence naming the most significant addition.) How does expanding behave? (Answer: shows additions one by one, each with its owner and any new gate it introduces.)
Design risk: Med Enables: Wave 2 & 3 content
06
Mobile / Field View (separate path)
The field supervisor's view is not a filtered version of the main flow — it is a separate product. Design it last (in the main build sequence) but design it from first principles, not as a responsive adaptation of the desktop flow. The constraints are: weak signal or offline, portrait phone, 3 actions maximum, bilingual. Build with the Branch Champion use case in mind — this is what the Champion shows a foreman on a job site, not what a foreman navigates alone.
Design risk: Low Separate build track
The fastest path to a shippable v1: Programme Dashboard + Gate Convergence Card + Persona Filter Bar, populated with Wave 0 and Wave 1 content only. Wave 2 and Wave 3 steps can be locked/greyed out in v1 — their content is authored but not yet accessible. This gives the CFO, Controller, and Payroll Manager a working flow for the first 6 months of the programme while Wave 2 and Wave 3 content is being authored and reviewed. It is better to ship Wave 0 + Wave 1 correctly than to ship all 3 waves partially.
